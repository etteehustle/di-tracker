Build a responsive web app MVP called “DI Tracker” for tracking and optimizing a Dual Investment financial system.

The user is building a personal financial tracking app focused mainly on Dual Investment (DI), especially for SOL/OKSOL, BTC, and ETH. The app is not a generic crypto portfolio tracker. It is a DI operating system + performance ledger + decision support tool.

The app must prioritize:
- Accurate DI capital tracking
- Active DI order tracking
- Settled DI order history
- Effective entry / exit calculation
- Cost basis tracking
- Multi-pocket DI capital management
- Forecasting 1-year DI growth
- Risk/performance evaluation before saving a new DI order
- Auditability

Base currency is always USDT.

====================================================
1. PRODUCT GOAL
====================================================

The user wants to know:

1. Total current capital inside the DI system.
2. Active DI orders with clear information:
   - exchange
   - product type
   - trading pair
   - subscribed amount
   - subscribed asset
   - APR
   - term rate
   - lock time
   - expected premium
   - if hit outcome
   - if not hit outcome
3. Current holding entry/cost basis if currently holding coin.
4. Total capital deposited into DI.
5. Profit/loss generated by DI.
6. Whether DI performance is positive or negative.
7. Historical order records for audit.
8. 1-year growth forecast based on DI order performance.
9. Ability to add new DI orders manually.
10. Ability to handle multiple DI pockets and multiple active orders.
11. Ability to merge pockets back into a core DI pocket.
12. Ability to track non-DI portfolio assets such as BTC or ETH holdings.
13. Current portfolio value using market price.
14. Manual portfolio adjustment with audit note.
15. Responsive access from desktop and mobile browser, 24/7.

====================================================
3. IMPORTANT BUSINESS RULES
====================================================

3.1 Base currency

All values, PnL, capital, forecast, and portfolio value must be shown in USDT.

3.2 SOL/OKSOL normalization

SOL and OKSOL are the same underlying exposure.

Rules:
- Display asset can be SOL or OKSOL.
- Underlying asset for both SOL and OKSOL is SOL.
- Use the same market price for SOL and OKSOL.
- Cost basis for OKSOL should be treated as SOL-equivalent.
- Portfolio allocation should group SOL and OKSOL under SOL exposure.

3.3 Current value

Current DI value must be mark-to-market.

Formula:
Current DI Value =
USDT balance
+ SOL/OKSOL amount × SOL price
+ BTC amount × BTC price
+ ETH amount × ETH price
+ active order conservative value

For active orders:
- Buy Low using USDT:
  - current value = subscribed USDT
  - expected premium should be shown separately as pending premium
- Sell High using coin:
  - current value = subscribed coin amount × current market price
  - expected premium should be shown separately as pending premium

Do not include unsettled pending premium inside the main conservative current value.

3.4 Forecast base

Forecast must use current mark-to-market DI value.

If the user is holding coin and coin dumps, the forecast should start from the current market value after the dump, not from cost basis and not from net deposited capital.

3.5 Cost basis method

Use Weighted Average Cost.

Formula:
Weighted Average Entry =
Total economic cost in USDT / Total asset amount

This applies especially to SOL/OKSOL, BTC, and ETH holdings.

3.6 Effective entry vs strike basis

The app must support two cost basis view modes:

Mode 1: Effective Entry
Formula:
Subscribed USDT / Total received coin including premium

Mode 2: Strike Basis
Formula:
Strike price, with premium shown separately

Default view should be Effective Entry because it reflects economic break-even better.

Example:
Subscribed = 3064 USDT
Received = 35.99708619 OKSOL
Strike = 86

Effective Entry = 3064 / 35.99708619 = 85.1180 USDT/SOL
Strike Basis = 86
Premium should be shown separately.

8. CAPITAL MOVEMENT
====================================================

The app must track capital deposits and withdrawals.

Deposit input:
- Deposit time
- USDT amount
- Note

Do not track VND amount or exchange rate.

Withdrawal must be classified clearly.

Withdrawal / movement types:

1. DI → Portfolio
Example:
The user withdraws 423 USDT from DI to buy BTC hold separately.

Effect:
- DI value decreases
- DI capital assigned decreases
- Portfolio value does not decrease
- Portfolio allocation changes
- DI forecast base decreases
- Total portfolio PnL should not be distorted

2. Portfolio → External
Example:
The user withdraws 500 USDT from the app/s Exchange to spend.

Effect:
- Portfolio value decreases
- If withdrawn from DI, DI value also decreases
- Must not be misread as trading loss
- Must require note

3. Internal pocket transfer
Example:
Move 500 USDT from Extra Pocket to Core Pocket.

Effect:
- Total DI value unchanged
- Total DI PnL unchanged
- Pocket balances change
- Must create audit/ledger entry

4. Manual adjustment
Used when actual balance differs from app because of dust, manual transfer, correction, split/merge, etc.

Manual adjustment must require note.

Manual adjustment must distinguish:
- DI balance adjustment
- Portfolio-only adjustment

This is important because BTC hold changes should not distort DI PnL.

====================================================
12. FORECAST
====================================================

The app must support exactly 2 forecast modes:

1. Settled Average
2. Blended

Do not implement Active Run-rate as a separate MVP mode.

12.1 Settled Average Forecast

This is the default forecast mode.

Data used:
- settled orders only

Purpose:
Shows forecast based on actual historical settled DI performance.

Formula:

Weighted Realized Daily Return =
Sum(realized DI yield from settled orders)
/
Sum(subscribed capital × actual lock days of settled orders)

1Y Forecast =
Current DI Value × (1 + Weighted Realized Daily Return) ^ 365

12.2 Blended Forecast

Data used:
- settled orders
- active orders

Purpose:
Shows forecast using both actual historical performance and expected premium from active orders.

Formula:

Blended Daily Return =
(
  Sum(realized DI yield from settled orders)
  + Sum(expected premium from active orders)
)
/
(
  Sum(settled subscribed capital × settled actual lock days)
  + Sum(active subscribed capital × active actual lock days)
)

1Y Forecast =
Current DI Value × (1 + Blended Daily Return) ^ 365

12.3 Forecast UI

Dashboard should show only simple output:

1-Year Forecast
Projected DI Value: X USDT
Confidence: Low / Medium / High
Mode: Settled Average or Blended

Do not show too many metrics on the dashboard.

Detailed forecast view can show:
- Mode
- Current DI value
- Projected 1Y value
- Daily run-rate
- Based on N settled orders
- Active order count if blended
- Data window
- Method: weighted by capital-days

12.4 Forecast confidence

MVP confidence rule:
- Low: less than 10 settled orders
- Medium: 10–30 settled orders
- High: more than 30 settled orders

Possible future improvement:
Confidence can later include:
- number of days of data
- market context diversity
- capital consistency
- outlier detection
- market regime coverage

12.5 Forecast must be a projection, not a promise.

27. IMPORTANT IMPLEMENTATION WARNING
====================================================

Do not build this as a simple portfolio tracker.

The core of this app is:
- DI order lifecycle
- DI ledger
- DI pocket accounting
- cost basis
- settlement
- weighted forecast
- audit

If any calculation is ambiguous, prioritize auditability and explicit ledger entries over hidden mutation.

Avoid making current balance the single source of truth. Use ledger entries as the source of truth and derive balances from them.

Do not count pocket merge as deposit or profit.

Do not count pending premium as settled profit.

Do not use cost basis as current DI value. Use market value.

Do not use net deposited capital as forecast base. Use current mark-to-market DI value.

Do not treat OKSOL as a separate economic asset from SOL.
